# Operations

## Install and CLI

`install.sh` is a one-command installer. Piped from curl it downloads a prebuilt
binary for the host (Linux `x86_64`/`aarch64`, static musl) and verifies its
SHA-256; if no release is published it falls back to fetching the source and
building it (auto-installing a Rust toolchain via rustup when `cargo` is
missing). Run from a checkout it builds the local tree. `--from-source` forces a
source build. Prebuilt binaries are published by `.github/workflows/release.yml`
on `v*` tags:

```bash
curl -fsSL https://raw.githubusercontent.com/CuriosityOS/AlturaProt/main/install.sh | sudo bash -s -- --start
```

System mode (default, needs root) installs to `/usr/local/bin`, creates the
`altura-prot` service user/group, writes config to `/etc/altura-prot`, and
installs the systemd unit; `--start` also enables and starts the service.
`--user` installs to `~/.local/bin` and `~/.config/altura-prot`. `--prefix PATH`
overrides the binary install root. The source repo and branch can be overridden
with the `ALTURA_PROT_REPO_URL` and `ALTURA_PROT_REPO_BRANCH` environment
variables.

### AI Power Detection step

After installing the binary, the installer runs an optional **AI Power
Detection** step. On a terminal (including `curl | bash`, which reads prompts
from `/dev/tty`) it asks whether to wire an AI provider for adaptive filter
generation and offers three choices:

1. **None** (default) — deterministic mitigation only.
2. **Subscription CLI** — wrap an agent CLI you already logged into (Claude,
   Codex, OpenCode, Cursor, Grok), the same approach as T3 Code. The installer
   marks which CLIs are on `PATH` and prints the vendor login command.
3. **API key** — OpenAI, Anthropic, Gemini, or OpenRouter; the key is stored in
   a `0600` secrets file owned by the service user.

When a provider is chosen the installer also copies the Python analyzer tools
(`codex_analyzer.py`, `ai_provider_cli.py`, `codexsdgate.py`) next to the
deployment (`/var/lib/altura-prot/tools` in system mode,
`~/.local/share/altura-prot/tools` for `--user`) and writes
`providers.json`/`secrets.json`. The proxy never calls AI on the request path;
this only configures the out-of-band CodexSDGate loop, which you can change
later with `ai_provider_cli.py`.

The step is skipped automatically when there is no terminal (e.g. CI) or with
`--non-interactive`, so agent/automation one-liners never hang. To configure it
non-interactively, pass `--ai PROVIDER` (one of `auto`, `none`, `codex`,
`claude`, `opencode`, `cursor`, `grok`, `openai`, `anthropic`, `gemini`,
`openrouter`) plus optional `--ai-model MODEL` and `--ai-key KEY`. With
`--ai auto` the installer selects the first installed agent CLI, else the first
provider whose API-key env var is set. For an explicit API-key provider, the key
is taken from the standard env var (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`,
`GEMINI_API_KEY`, `OPENROUTER_API_KEY`) when `--ai-key` is omitted:

```bash
# agent-friendly: install and wire whatever AI is already available
curl -fsSL https://raw.githubusercontent.com/CuriosityOS/AlturaProt/main/install.sh \
  | bash -s -- --user --ai auto --non-interactive

# user install that wires Gemini in one shot (key from $GEMINI_API_KEY)
GEMINI_API_KEY=sk-... curl -fsSL .../install.sh \
  | bash -s -- --user --ai gemini --non-interactive

# system install that selects the already-logged-in Claude CLI
sudo ./install.sh --start --ai claude --non-interactive
```

There is no capability difference between a subscription CLI and an API key:
both drive the same analyzer pipeline and produce the same adaptive filters.

### Analyzer timer (optional, system mode)

By default you run the analyzer yourself (the installer prints the command). To
have it run automatically, install the systemd timer with `--ai-timer` (or answer
yes to the interactive prompt). It installs `altura-prot-analyzer.service`
(a `oneshot` that runs `codexsdgate.py --once` as the `altura-prot` user) and
`altura-prot-analyzer.timer`, and enables the timer:

```bash
sudo ./install.sh --start --ai auto --ai-timer --ai-interval 120 --ai-threshold 20
systemctl status altura-prot-analyzer.timer
journalctl -u altura-prot-analyzer.service     # see each run
```

The timer polls every `--ai-interval` seconds (default 120), but because the
analyzer is threshold-gated the AI provider is only actually called when a batch
crosses `--ai-threshold` real attack events (default 20) — so frequent polling is
cheap and the AI only engages during real attacks. The analyzer unit deliberately
omits `MemoryDenyWriteExecute` (unlike the proxy unit) because the Node/V8-based
agent CLIs need W^X JIT pages.

For an **API-key** provider this works out of the box (the key is read from the
service user's `secrets.json`). For a **subscription CLI** provider the timer runs
as the `altura-prot` user, which must be logged into that CLI; until it is, the
analyzer falls back to the deterministic generator. The installer prints the exact
`sudo -u altura-prot … login` command. User-mode installs do not get a timer — run
the analyzer via cron or `systemd --user`.

The `altura-prot` binary is also a management CLI:

```bash
altura-prot init [--system]                        # create config dir + default config
altura-prot validate                               # validate the active config file
altura-prot config show                            # print the resolved config as JSON
altura-prot config get http.limits.per_ip_rps      # read one value by dot path
altura-prot config set http.admin_token <secret>   # set one value (validated before write)
altura-prot run [--config PATH]                    # start the proxy
altura-prot status                                  # systemd or process status
```

`config set` is atomic and validated: it writes a temp file, runs full config
validation, and renames into place only on success, so a rejected value never
replaces a working config. The active config path resolves from `--config`,
then `$ALTURA_PROT_CONFIG`, `/etc/altura-prot/config.json`,
`~/.config/altura-prot/config.json`, and finally `configs/example.json`. The
legacy `altura-prot --config PATH` form (no subcommand) still runs the proxy.

## Metrics

Health:

```bash
curl http://127.0.0.1:8080/__altura/health
```

Metrics:

```bash
curl -H 'x-altura-admin-token: change-me' http://127.0.0.1:8080/__altura/metrics
```

`GET /__altura/health` and `GET /__altura/metrics` are subject to the HTTP request token buckets, including normalized request-signature limits. Admin health, authorized metrics, and forbidden metrics responses use `Cache-Control: no-store` so intermediaries do not replay stale control-plane state, and generated admin responses close the downstream connection so body-bearing control-plane requests cannot be followed on the same socket. Metrics fail closed with `403` unless `http.admin_token` is configured and exactly one matching `x-altura-admin-token` header is supplied.

## Blackhole Guidance

Use application filters for L7 selectivity and uptime. Use upstream blackholing or scrubbing when traffic volume saturates the link or host before the proxy can make useful decisions.

Blackholing is cheaper than host firewall processing, but it drops all matched traffic. For public production use, pair AlturaProt with a provider-level control such as BGP blackhole, scrubbing center, anycast/CDN, or an upstream router ACL.

See [Edge Protection](EDGE_PROTECTION.md) for the provider, nftables, and sysctl production shape.

## Hardening Checklist

- Run behind TLS termination or add TLS support before internet exposure.
- Put public deployments behind provider/CDN/scrubbing protection for volumetric floods.
- Validate and tune `ops/nftables/altura-prot-edge.nft`, `ops/sysctl.d/99-altura-prot-ddos.conf`, and `ops/systemd/altura-prot.service` on the target Linux host before installing; run `python3 tools/validate_edge_templates.py --config configs/example.json` as the non-mutating preflight. The preflight fails when a non-loopback AlturaProt HTTP/TCP listener port is missing from the nftables `protected_tcp_ports` set, when the protected-port connection-count threshold is looser than a public userspace per-IP cap, when the raw UDP drop for protected TCP service ports is missing or omits `meta l4proto udp`, when generic dual-stack protected-port TCP backstops omit `meta l4proto tcp`, when dynamic nftables SYN-rate sets or connection-limit sets omit explicit positive `size` bounds, when SYN-rate sets omit finite timeout support, when IPv6 protected-port SYN and connection-count backstops are keyed by exact source address instead of the template's `/64` prefix mask, when IPv6 TCP/ICMPv6 flood backstops use `ip6 nexthdr` instead of extension-header-safe `meta l4proto`, when essential ICMPv4 or ICMPv6 control messages are not exempted before the generic ICMP flood drops, or when the systemd unit drops required service-manager backstops.
- Set `runtime.min_nofile` high enough for configured HTTP/TCP connection caps and upstream sockets, and set the service-manager hard limit first when needed. For systemd units, keep `LimitNOFILE=` at or above both the configured floor and the socket budget estimated from listeners, downstream connections, upstream in-flight requests, idle upstream pool, and TCP socket pairs. Startup fails if the final soft NOFILE limit cannot cover the configured socket budget.
- Install AlturaProt as a non-root service user. The shipped systemd template uses `User=altura-prot`, `Group=altura-prot`, finite `TasksMax`, `MemoryHigh`, and `MemoryMax`, restart-storm limits, `NoNewPrivileges`, write access limited to `/var/lib/altura-prot` and `/var/log/altura-prot`, and only `CAP_NET_BIND_SERVICE` for privileged port binding. Tune memory ceilings from production traffic and host RAM, but keep them finite so a userspace memory leak or flood-amplified state path is contained to the unit.
- Keep the `--config` input as a regular JSON file below the 1 MiB startup read cap. Startup rejects oversized or non-regular config paths before JSON parsing so a bad deploy artifact cannot turn service start into an unbounded memory/read path.
- Configure `http.upstream` as a plain `http://` URI with a host authority. Startup rejects missing schemes, `https://` upstreams, URI userinfo, and query strings because the current upstream client uses Hyper's plain HTTP connector; add and validate an explicit TLS connector before accepting HTTPS origins.
- Configure `tcp[].upstream` as a bare `host:port` authority, using bracketed IPv6 such as `[::1]:9001` when needed. Startup rejects URL schemes, URI userinfo, path/query/fragment suffixes, missing or nonnumeric ports, port `0`, whitespace, and control characters so a bad TCP upstream typo cannot leave the proxy accepting clients only to fail every upstream connect on the request path.
- Treat `0` as the only explicit disable value for HTTP/TCP rate buckets and optional minimum data-rate byte floors. Negative or non-finite values in `http.limits.*_rps`, HTTP connection-open rate caps, and TCP connection-open rate caps fail startup with the exact config path so a typo cannot silently remove a DDoS guardrail.
- Keep `http.allowed_methods` non-empty and small. Startup rejects empty lists, invalid HTTP method tokens, duplicate entries, more than 16 configured methods, method tokens longer than 32 bytes, and unsupported tunnel/diagnostic method names (`CONNECT`, `TRACE`, and `TRACK`), so a bad deploy cannot turn every request into a method denial with an empty/malformed `Allow` header, produce oversized generated `405` responses, or opt the reverse proxy into unsupported forwarding semantics.
- Leave `http.allow_method_override_headers` disabled unless the origin intentionally uses legacy override headers (`X-HTTP-Method`, `X-HTTP-Method-Override`, or `X-Method-Override`) and has its own method authorization. The default rejects those headers before origin work so `http.allowed_methods` cannot be bypassed by sending an allowed outer method with a different semantic method in a header.
- Leave `http.allowed_hosts` empty only when the Host allowlist is intentionally disabled. When configured, startup rejects blank entries, whitespace/control characters, invalid HTTP authorities, URI userinfo, wildcard-looking entries, duplicate entries, more than 128 hosts, entries longer than 255 bytes, and entries above `http.max_host_bytes`, so a bad deploy cannot create surprising Host matching or excessive per-request allowlist scans.
- Keep DDoS-critical resource ceilings greater than zero, except where a field explicitly documents `0` as a disable mode. Startup rejects zero connection caps, in-flight caps, limiter-state caps, parser metadata caps, `http.max_ranges`, body-size caps, trailer caps, forwarded-header caps, listener backlogs, accept-shard counts, static/runtime filter caps, adaptive detector windows, event-log queue size, event-log rotation size, event-log backup count, and active timeout/flush/cooldown/grace windows so a typo cannot remove the finite bounds that protect CPU, memory, sockets, file descriptors, disk, analyzer evidence, filter hot-path CPU, and origin concurrency. Keep `http.max_header_bytes` and `http.upstream_max_header_bytes` between `8192` and `262144` bytes: the lower bound matches Hyper's HTTP/1 parser buffer floor, and the upper bound keeps accidental config from admitting excessive per-connection parser buffers. The raw connection-opening precheck treats `http.max_header_bytes` as a hard cap even when the header terminator arrives in the same read after the cap, and scans only the newly appended suffix plus overlap. Keep `http.max_headers` and `http.upstream_max_headers` between `1` and `1024`; the default `100` should cover normal traffic, and higher values should be based on measured proxy/origin metadata needs. Keep `http.max_body_bytes` and `http.max_upstream_body_bytes` between `1` and `1073741824` bytes; defaults are `10485760` and `104857600`, and values near the ceiling should be reserved for measured upload/download requirements because they let one admitted stream consume more proxy and origin bandwidth before rejection. Keep `http.header_read_timeout_ms` between `1` and `60000`; the default `5000` is the intended internet-facing slowloris posture, and values near the ceiling should be justified by measured client/network behavior. Keep `http.downstream_write_timeout_ms`, `http.request_body_idle_timeout_ms`, `http.upstream_body_idle_timeout_ms`, `http.request_body_min_rate_grace_ms`, and `http.upstream_body_min_rate_grace_ms` between `1` and `60000`; their defaults are `15000`, `10000`, `15000`, `10000`, and `10000`, and values near the ceiling should be reserved for measured slow-client or slow-origin behavior because they keep request, response, and upstream permits alive longer under slow-reader or slow-body pressure. Keep `http.upstream_connect_timeout_ms` and `tcp[].connect_timeout_ms` between `1` and `60000`; the shared default is `1000`, and values near the ceiling should be reserved for measured private-network behavior because they hold origin or TCP connection permits during failed upstream connection attempts. Keep `http.upstream_timeout_ms` between `1` and `60000`; the default `15000` bounds origin response-header waits, and values near the ceiling should be reserved for measured origin behavior because they hold upstream in-flight permits longer during stuck-origin conditions. Keep `http.upstream_pool_idle_timeout_ms` between `1` and `60000` and keep `http.upstream_pool_max_idle_per_host` between `0` and `4096`; the defaults are `30000` and `256`, and `0` should be used only when intentionally disabling idle origin pooling. Keep `http.max_connection_duration_seconds` and `tcp[].max_connection_duration_seconds` between `1` and `3600`; the defaults are `120` and `300`, and values near the ceiling should be reserved for measured long-lived traffic because they keep connection permits alive longer under connection pressure. Keep `http.max_requests_per_connection` between `1` and `10000`; the default is `1000`, and values near the ceiling should be reserved for measured keep-alive reuse because too many requests on one persistent connection can retain per-connection state for excessive work. Keep `filters.max_runtime_file_bytes` between `1` and `16777216`, and keep `filters.max_runtime_filters` plus `filters.max_static_filters` between `1` and `8192`; larger rule sets should be reduced or split upstream because every loaded rule participates in request-path filtering. Keep `adaptive.event_log_max_bytes` between `1` and `1073741824`, `adaptive.max_signature_windows` plus `adaptive.max_path_shape_windows` between `1` and `262144`, and `adaptive.event_log_queue_capacity` between `1` and `8192`; request workers drop attack events instead of blocking when this queue fills, so large values should be based on measured writer latency and memory budget. Keep `adaptive.event_log_backup_count` between `1` and `128`; each rotation scans numbered backups, so larger long-term retention belongs in external log archival instead of the attack-event writer's hot rotation path. Keep `http.limits.max_tracked_ips` and `tcp[].limits.max_tracked_ips` between `1` and `1048576`, and keep `http.limits.max_tracked_signatures` plus `http.limits.max_tracked_path_shapes` between `1` and `262144`; HTTP rate buckets evict stale state but deny new active keys when a shard is full, while connection and in-flight limiters evict inactive state and deny new entries when all tracked entries in a shard are active. This prevents high-cardinality rotation from resetting hot rate buckets while still keeping limiter maps inside startup-bounded memory.
- Limiter cleanup probes are capped per insert attempt, so a new distinct key cannot force a full large-shard scan while holding the limiter mutex. Normalized client-prefix keys are hash-sharded after prefix normalization, so IPv4 network keys ending in `.0` are distributed by the key hash instead of collapsing into one low-bit shard. Global request and connection-open token buckets are checked in short critical sections and are not held across per-client shard cleanup. Per-client request/TCP/HTTP connection-open tokens are consumed only after the other relevant global and per-client gates also admit the attempt, and obviously globally saturated connection or in-flight gates reject before creating avoidable per-client state. If limiter state stays at capacity, first lower cardinality or raise the relevant cap from memory and route-cardinality measurements instead of relying on cleanup to absorb attacker-paced churn.
- Keep `adaptive.activation_ttl_seconds` and per-filter `ttl_seconds` between `1` and `86400`. Keep static/runtime filters narrow: startup and runtime reload reject root-wide `path_prefix: "/"` rules because they would block the whole site after one bad config or analyzer output.
- Keep `http.request_body_min_rate_bytes_per_second`, `http.upstream_body_min_rate_bytes_per_second`, `tcp[].downstream_min_rate_bytes_per_second`, and `tcp[].upstream_min_rate_bytes_per_second` between `0` and `1048576`. The HTTP and TCP defaults are `512` B/s after the grace window; use `0` only as an explicit opt-out for measured quiet protocols. Enabled guards check recent post-grace byte windows measured from the grace deadline once post-grace bytes arrive, not lifetime connection averages, so an initial fast payload cannot bank unlimited credit or a free first post-grace drip for later slow-drip behavior. Values near the ceiling should require traffic measurements because an excessive minimum rate can shed valid slow clients or origins during congestion.
- Keep `http.max_header_line_bytes` and `http.upstream_max_header_line_bytes` positive, no higher than `262144`, and no higher than their respective total header byte caps. The default `8192` mirrors common proxy/server per-field defaults and prevents one large cookie, user-agent, or origin response field from occupying the whole configured header buffer. Increase only for measured legitimate metadata such as large authentication headers, and raise the total header byte cap deliberately when a larger per-field cap is justified.
- Keep request metadata caps finite: `http.max_host_bytes` no higher than `1024`, `http.max_uri_bytes` and `http.max_query_bytes` no higher than `65536`, `http.max_query_pairs` no higher than `8192`, and `http.max_path_segments` no higher than `4096`. Defaults are much smaller (`255`, `8192`, `4096`, `128`, and `64`) and should cover normal browser/API traffic. For absolute-form request targets, `http.max_uri_bytes` includes scheme, `://`, authority, and path/query bytes, not only the origin-form path/query. Higher values should be based on measured route and query requirements because request-target parsing happens before signature, filter, rate-limit, or upstream work.
- Keep trailer and trusted-forwarded-header caps finite: `http.max_trailer_bytes` and `http.upstream_max_trailer_bytes` no higher than `262144`, `http.max_trailers` and `http.upstream_max_trailers` no higher than `1024`, `http.client_ip.max_forwarded_for_bytes` no higher than `16384`, and `http.client_ip.max_forwarded_for_hops` no higher than `256`. Defaults are `8192` trailer bytes, `32` trailers, `1024` trusted forwarded bytes, and `32` trusted forwarded hops; values near the ceilings should require measured trailer metadata or proxy-chain depth.
- Keep `http.upstream_failure_threshold` between `1` and `1024`, and keep `http.upstream_failure_open_ms` between `1` and `300000`. The defaults are `8` and `1000`; values near the ceilings should be based on measured origin restart/failure behavior because they delay local shedding or keep origin-bound traffic shed longer after failures.
- Keep `http.admin_path_prefix` as an absolute non-root path prefix such as `/__altura`. Startup rejects empty, root, relative, query-bearing, fragment-bearing, whitespace, or control-character prefixes so health/metrics endpoints cannot be accidentally moved to `/health`/`/metrics` or made unreachable by typo.
- Leave `http.admin_token` unset only when metrics should fail closed with `403`; when configured, use a non-empty, non-blank, trimmed secret value without control characters and no more than 256 bytes. Startup rejects blank, padded, or oversized tokens so metrics auth cannot be weakened by an accidental empty shared secret.
- Put metrics behind an admin token and private network; do not run public metrics without `http.admin_token`.
- Keep default listeners bound to explicit addresses.
- Configure `http.client_ip.trusted_proxies` before trusting `X-Forwarded-For`; never trust forwarded, CDN, or proxy client-IP headers from arbitrary internet peers. AlturaProt validates `http.client_ip.header` as a non-empty, trimmed, bounded HTTP field name and rejects malformed trusted-proxy lists at config load: entries must be non-empty, trimmed, free of whitespace/control characters, no more than 64 bytes each, no more than 128 total, duplicate-free, and valid IP/CIDR ranges. Startup fails when a non-loopback HTTP listener trusts all peers with ranges such as `0.0.0.0/0` or `::/0`. The runtime resolver relies on validated config and does not silently fall back to a weaker default header or drop invalid proxy entries. AlturaProt overwrites spoofable forwarded metadata, strips common client-IP identity headers, and consumes standard plus `Connection`-listed hop-by-hop headers before forwarding request or response messages. Duplicate trusted `X-Forwarded-For` field lines are canonicalized into one ordered comma-separated chain before upstream forwarding when XFF is the configured identity header, so the origin-visible chain matches the values used for client-IP resolution. If a trusted deployment uses a different configured identity header such as `CF-Connecting-IP`, that header must be a single non-empty IP value; duplicate field lines and comma-list values return `400` and increment `altura_http_forwarded_rejected`. Accepted custom identity headers do not preserve inbound XFF; AlturaProt synthesizes a clean `resolved-client, peer` XFF chain and still sets `X-Real-IP` to the resolved client. Keep `http.client_ip.max_forwarded_for_bytes` and `http.client_ip.max_forwarded_for_hops` finite so trusted-forwarded parsing has bounded CPU and allocation cost. Keep `http.limits.trusted_proxy_rps`, `http.limits.trusted_proxy_burst`, and `http.limits.trusted_proxy_max_in_flight_requests` finite for trusted edge deployments so one proxy peer cannot carry unlimited rotated forwarded identities or monopolize origin concurrency.
- Tune `http.limits.ipv4_prefix_len` and `http.limits.ipv6_prefix_len` for the client-identity granularity you want on HTTP per-client request, connection-open, and upstream in-flight limits. Defaults are IPv4 `/32` and IPv6 `/64`, which keeps IPv4 exact while grouping IPv6 privacy-address or same-subnet rotation into one protected bucket. Startup rejects prefix lengths outside `1..32` for IPv4 and `1..128` for IPv6. Trusted-proxy aggregate limits intentionally remain exact immediate-peer caps; use `trusted_proxy_rps`, `trusted_proxy_burst`, and `trusted_proxy_max_in_flight_requests` for the edge-hop aggregate budget.
- Keep the host-edge nftables IPv6 `/64` SYN and connection-count masks aligned with the userspace default unless you intentionally choose a different IPv6 client-prefix policy. Changing `http.limits.ipv6_prefix_len` or `tcp[].limits.ipv6_prefix_len` away from `/64` should trigger a matching provider/host-edge review, because the shipped nftables template is intentionally hardcoded to `/64`.
- Keep generic protected-port TCP nftables predicates and IPv6-specific predicates on `meta l4proto`, not bare transport matches or `ip6 nexthdr`. `ip6 nexthdr` only sees the immediate IPv6 next-header field, while `meta l4proto` follows extension headers to the real TCP or ICMPv6 transport protocol.
- Keep the nftables `tcp4_syn_rate` and `tcp6_syn_rate` dynamic sets explicitly size-bounded and timeout-bounded. The shipped template uses `size 65535` and `timeout 10s`; increase only after sizing kernel memory and expected protected-port source cardinality.
- Keep the nftables `tcp4_connlimit` and `tcp6_connlimit` dynamic sets explicitly size-bounded. The shipped template uses `size 65535`; increase only after sizing kernel memory and expected protected-port client cardinality. Do not add timeouts to the `ct count` connlimit sets unless the target nftables/kernel combination explicitly supports that policy.
- Keep the ICMP control exemptions before the generic ICMP flood drops. The shipped template exempts IPv4 destination unreachable, time exceeded, and parameter problem, plus IPv6 destination unreachable, packet too big, time exceeded, parameter problem, router solicitation/advertisement, and neighbor solicitation/advertisement, so PMTUD and IPv6 Neighbor Discovery remain functional while other ICMP remains rate-backed.
- Keep fragment reassembly sysctls bounded. The shipped profile caps IPv4/IPv6 fragment memory at 4 MiB, keeps IPv6 low/high thresholds ordered, retains incomplete fragment queues for 15 seconds, and leaves IPv4 `ipfrag_max_dist` at 64. Raise these only after confirming the host needs fragmented traffic and has memory headroom; AlturaProt itself should rely on MSS/PMTUD, not IP fragmentation.
- Tune `runtime.min_nofile`, `runtime.shutdown_grace_ms`, `allowed_methods`, `allowed_hosts`, `max_host_bytes`, `listen_backlog`, `accept_shards`, `per_ip_rps`, `http.limits.ipv4_prefix_len`, `http.limits.ipv6_prefix_len`, `signature_rps`, `path_shape_rps`, `trusted_proxy_rps`, `trusted_proxy_max_in_flight_requests`, `global_rps`, HTTP connection-open rate caps, active connection caps, in-flight upstream request caps, upstream connect timeout (`upstream_connect_timeout_ms`), upstream failure circuit settings (`upstream_failure_threshold`, `upstream_failure_open_ms`), upstream idle-pool caps, upstream response header caps, request/response body limits, `http.allow_chunked_request_bodies`, `http.allow_compressed_request_bodies`, `http.allow_expect_continue`, `http.max_ranges`, `http.forward_accept_encoding`, trusted-forwarded-header caps (`http.client_ip.max_forwarded_for_bytes`, `http.client_ip.max_forwarded_for_hops`), request/upstream body minimum data rates and grace windows, request trailer policy (`forward_request_trailers`, `max_trailer_bytes`, `max_trailers`), upstream response trailer policy (`forward_response_trailers`, `upstream_max_trailer_bytes`, `upstream_max_trailers`), `downstream_write_timeout_ms`, request-target/query/path caps, header byte/field/count caps, header/idle-upload/upstream-response idle timeouts, static/runtime filter caps (`filters.max_static_filters`, `filters.max_runtime_file_bytes`, `filters.max_runtime_filters`), adaptive thresholds, `adaptive.max_signature_windows`, `adaptive.max_path_shape_windows`, `adaptive.event_log_queue_capacity`, `adaptive.event_log_flush_interval_ms`, `adaptive.event_log_max_bytes`, and `adaptive.event_log_backup_count` from real traffic, proxy topology, memory budgets, and disk budgets. `runtime.shutdown_grace_ms` defaults to `2000`; use `0` only when intentionally favoring fast local benchmark child exit over a final drain window. Keep large learned rule sets in the bounded runtime filter file rather than embedding them in startup config; the config file itself is capped at 1 MiB. Keep `http.max_body_bytes` and `http.max_upstream_body_bytes` positive and no higher than `1073741824` so request and response body guards remain finite even after a bad deploy. Keep `http.upstream_pool_idle_timeout_ms` positive and no higher than `60000`, and keep `http.upstream_pool_max_idle_per_host` no higher than `4096`; use `0` for that pool-size field only when intentionally disabling idle origin connection reuse. Keep `http.max_requests_per_connection` positive and no higher than `10000` so the per-connection request cap cannot be configured into effectively unlimited persistent-connection work. Keep `upstream_connect_timeout_ms` positive, no higher than `60000`, and below `upstream_timeout_ms` when you want connect failures to surface before the broader upstream header wait budget. Keep `downstream_write_timeout_ms`, request/upstream body idle timeouts, and request/upstream body minimum-rate grace windows positive and no higher than `60000` so slow-reader and slow-body paths stay bounded even after a bad deploy. Keep `signature_rps` and `path_shape_rps` below `global_rps` when you want one normalized request family to shed before it burns the service-wide budget, and set `max_tracked_signatures` plus `max_tracked_path_shapes` from memory budget and route-cardinality expectations. Set `upstream_failure_threshold` high enough to avoid tripping on isolated origin jitter and set `upstream_failure_open_ms` short enough to allow recovery probes; open-circuit requests return `503` with `Retry-After: 1`, `Cache-Control: no-store`, and `Connection: close` without attempting another upstream connection. Oversized declared upstream `Content-Length` values are rejected before upstream success is recorded, returning a generated `502` with `Cache-Control: no-store` and `Connection: close`. Other generated upstream `502` and `504` responses return `Cache-Control: no-store` and `Connection: close` so transient origin failures are not cached and clients do not reuse downstream sockets after proxy-owned upstream failure paths. Upstream in-flight rejections return `503` with `Retry-After: 1`, `Cache-Control: no-store`, and `Connection: close`; tune the in-flight caps from origin capacity so this path protects the origin without becoming a normal steady-state response. Per-connection request-cap rejections return `429` with `Retry-After: 1`, `Cache-Control: no-store`, and `Connection: close`; tune `max_requests_per_connection` low enough to bound persistent-connection abuse but high enough for legitimate browser reuse. Request-body idle and minimum-rate timeouts return `408` with `Cache-Control: no-store` and `Connection: close` so slow upload clients reconnect instead of reusing a timed-out request stream and intermediaries do not store synthetic timeout responses. Initial raw oversized-header rejections return `431` with `Cache-Control: no-store` and `Connection: close` before Hyper parsing so header-byte and header-field floods get a precise non-storeable denial. Request-framing rejections return `400` with `Cache-Control: no-store` and `Connection: close` so malformed or policy-rejected body framing cannot be followed by another request on the same downstream connection. Unsupported request content codings return `415` with `Accept-Encoding: identity`, `Cache-Control: no-store`, and `Connection: close` so clients do not reuse a connection with an unread encoded upload body. Oversized request-body rejections return `413` with `Cache-Control: no-store` and `Connection: close` so clients do not attempt to reuse a connection carrying rejected or unread body bytes. Oversized forwarded request-trailer rejections return `431` with `Cache-Control: no-store` and `Connection: close` so opt-in trailer forwarding cannot leave a reusable connection after rejected end-of-message metadata. Request-target pressure rejections return `414` with `Cache-Control: no-store` and `Connection: close` so attack-shaped long URI/query/path responses are not stored heuristically by intermediaries and body-bearing rejects cannot be followed on the same downstream connection. Method-guard rejections return `405` with `Allow`, `Cache-Control: no-store`, and `Connection: close` so disallowed-method guard responses do not become stale heuristic cache entries and body-bearing rejects cannot be followed on the same downstream connection. Other generated early-deny responses emitted before request content is consumed, including filter blocks and expect/range/host/forwarded validation failures, also return `Cache-Control: no-store` and close the downstream connection. Enable `allow_chunked_request_bodies` only for origins that truly need unknown-length HTTP/1.1 uploads; `Content-Length` uploads can be rejected by size before upstream work, while chunked bodies rely on streaming guards after admission. Do not use `max_connections: 0` with `runtime.min_nofile`; the startup capacity check requires finite HTTP/TCP connection ceilings.
- Keep the path-shape-scoped passive upstream failure circuit finite: `http.upstream_failure_threshold` must stay `1..=1024`, and `http.upstream_failure_open_ms` must stay `1..=300000`, so a bad deploy cannot make a shape circuit effectively unreachable or hold local origin shedding for hours. Circuit state is keyed by normalized path shape and bounded by `http.limits.max_tracked_path_shapes`; repeated failures in one route family should not fail-fast unrelated route families. When the circuit table is full, AlturaProt preserves open circuit entries, reclaims closed or expired entries, and skips tracking a new failing shape if every probed entry is still open.
- Initial raw header-count rejections return `431` with `Cache-Control: no-store` and `Connection: close` before Hyper parsing, so header-spray requests are visible and use the same non-storeable denial contract as header-byte pressure.
- Initial raw header read timeouts return `408` with `Cache-Control: no-store` and `Connection: close` before Hyper parsing, so slow connection-opening header drips free connection permits with an explicit non-storeable denial.
- Initial raw request-target pressure rejections return `414` with `Cache-Control: no-store` and `Connection: close` before Hyper parsing, so URI, query, and path-shape floods are denied before request construction and before signature/filter/upstream work.
- Leave `http.downstream_keep_alive` disabled for internet-facing services unless measured latency/throughput requirements justify persistent client connections and you have compensating edge controls.
- For reverse-proxy deployments, set `allowed_hosts` so both regular `Host` headers and absolute-form request-target authorities are constrained to expected service names. Absolute-form request targets with non-HTTP schemes are rejected before origin work; keep upstream-specific protocols behind explicit TCP proxy entries instead of relying on HTTP request-target schemes.
- The nftables template drops UDP packets directed at `protected_tcp_ports` before conntrack because AlturaProt listeners are TCP-only. Narrow or remove that rule if the same host intentionally serves UDP/QUIC on one of those ports.
- Monitor `altura_http_framing_rejected` for malformed `Content-Length`/`Transfer-Encoding` requests that reach the application-level guard; parser-rejected or parser-normalized HTTP/1 cases can occur before this counter increments.
- Monitor `altura_http_header_line_rejected` for parsed request headers rejected because one field exceeded `http.max_header_line_bytes`; on internet-facing services, sustained increases usually indicate oversized cookie/user-agent pressure or an opt-in keep-alive client trying to bypass the raw first-request precheck.

- Monitor `altura_http_initial_headers_too_many` for connection-opening requests rejected by the raw HTTP/1 header-count cap before Hyper parsing.
- Monitor `altura_http_initial_header_timeouts` for slow connection-opening requests rejected by the raw HTTP/1 header read timeout before Hyper parsing.
- Monitor `altura_http_initial_request_target_rejected` for connection-opening requests rejected by the raw HTTP/1 request-target caps before Hyper parsing.
- Monitor `altura_http_content_encoding_rejected` for compressed request bodies rejected before reaching the origin. Keep `http.allow_compressed_request_bodies` disabled unless the origin explicitly needs compressed uploads and has decoded-size, decompression-ratio, CPU, and memory limits of its own.
- Monitor `altura_http_expect_rejected` for rejected `Expect` requests. Keep `http.allow_expect_continue` disabled unless the origin and client population require `100-continue` upload negotiation and the body limits/min-rate guards have been tuned for those uploads.

- Monitor `altura_http_forwarded_sanitized` for requests where spoofable forwarded metadata or CDN/proxy client-IP headers were removed before upstream proxying.

- Monitor `altura_http_connections_rejected` for HTTP connection-open churn rejected before request parsing. Sustained increases can mean one edge hop is carrying a spoofed/rotating-identity burst or that the cap is below legitimate aggregate traffic for that proxy.
- Monitor `altura_http_trusted_proxy_in_flight_rejected` for trusted proxy peers that exceeded their aggregate upstream concurrency cap after forwarded client-IP resolution. Sustained increases can mean one edge hop is carrying slow origin-consuming traffic with rotated identities, or that `trusted_proxy_max_in_flight_requests` is below legitimate concurrency for that proxy.

Sustained increases mean one or more route families are repeatedly failing or timing out; check `altura_http_upstream_errors`, `altura_http_upstream_timeouts`, origin health by route family, and whether `upstream_failure_threshold` or `upstream_failure_open_ms` are too aggressive for normal deploy/restart behavior.
- Monitor `altura_http_request_trailers_dropped`, `altura_http_request_trailers_rejected`, `altura_http_upstream_trailers_dropped`, and `altura_http_upstream_trailers_rejected` when enabling trailer forwarding or when debugging clients/origins that rely on trailing headers.

Sustained counts at capacity mean high-cardinality traffic is forcing the detector to preserve recent windows and stop admitting some new keys until idle windows age out; raise the corresponding cap only when memory budget allows it and detection fidelity requires broader retention.

Sustained signature limiting means one normalized method/path/query/user-agent/accept shape is being shed before global limits; sustained path-shape limiting means a route family is hot even if attackers rotate long tokens, high-confidence short tokens, or many distinct short lowercase siblings under one parent route. Raise `signature_rps` or `path_shape_rps` only after confirming the origin can handle that shape and benign traffic really needs the headroom.
- AlturaProt's deterministic HTTP request-rate limiters return `429` with `Retry-After: 1` and `Cache-Control: no-store`. Clients and synthetic monitors should honor `Retry-After` instead of retrying immediately; attack traffic is still dropped by the limiter if it ignores the hint. Intermediaries should not store the generated 429 responses.
- Monitor `altura_event_log_dropped`; any sustained increase means the adaptive event-log path is preserving request availability by dropping telemetry. The counter covers both request-side full-queue drops and writer-side discard after serialization, open, rotate, write, or newline failure. JSON serialization, file writes, idle flushes, and rotation are worker-side, Unix event-log files are created/fixed to owner-only permissions, and a positive flush interval still drains batched events after idle timeouts. Increase `adaptive.event_log_queue_capacity` within the `8192` startup ceiling, reduce event volume, or move the event log to faster storage if analyzer fidelity matters during that traffic pattern.

- Keep static and learned runtime filters narrow. Keep `filters.max_static_filters`, `filters.max_runtime_file_bytes`, and `filters.max_runtime_filters` finite so bad config or analyzer output cannot make the proxy parse oversized filter files or evaluate an unbounded rule set on every request.
