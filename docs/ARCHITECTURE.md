# Architecture

AlturaProt has two loops:

1. Hot path: Rust reverse proxy accepts HTTP/TCP traffic, applies limits and filters, proxies allowed requests, and emits compact telemetry through a bounded nonblocking queue whose writer thread handles JSON serialization, flush cadence, and byte-cap rotation.
2. Control path: CodexSDGate reads telemetry and writes constrained JSON filters that the proxy reloads behind file-size, file-type, and filter-count caps.

The hot path must remain deterministic. AI providers never run inline with client requests because model latency, provider outages, and token cost would make availability worse during an attack.

## Runtime Preflight

- Validate the config file and parsed DDoS-critical settings before listeners start. The `--config` path must be a regular file and is read behind a 1 MiB cap before JSON deserialization. HTTP listen addresses must be valid socket addresses, and `http.upstream` must be a plain `http://` URI with a host authority, no URI userinfo, and no query string. TCP listen addresses must be valid socket addresses, and `tcp[].upstream` must be a `host:port` authority with no URL scheme, URI userinfo, path, query, fragment, whitespace, or control characters, and with a numeric nonzero port. Negative or non-finite rate values fail startup with the exact config path; `0` remains the explicit disable value for a specific rate bucket. Connection caps, in-flight caps, limiter-state caps, parser metadata caps, trailer caps, forwarded-header caps, body-size caps, timeouts, listener backlogs, bounded accept-shard counts, upstream failure circuit settings, static/runtime filter caps, adaptive detector windows, event-log queue size, event-log rotation size, event-log backup count, and event-log flush/cooldown windows must be greater than zero. HTTP request and upstream response body-size caps are also capped at `1073741824` bytes at startup so a bad config cannot turn the body guards into effectively unbounded byte streams. HTTP request/upstream body and raw TCP optional minimum data-rate byte floors are capped at `1048576` B/s so a bad config cannot shed valid slow clients or origins by typo. HTTP request-target, Host, trailer, and trusted-forwarded-header metadata caps are startup-capped so a bad config cannot turn request classification, trailer handling, or trusted proxy identity parsing into excessive request-path work. HTTP request and upstream response header field-line caps are positive, capped at `262144`, and must not exceed their respective total header-buffer caps, so one oversized field cannot consume the whole metadata budget. HTTP per-connection request counts are capped at `10000` so an opt-in persistent downstream connection cannot be configured into effectively unlimited per-socket request work. Upstream failure circuit thresholds are capped at `1024` consecutive failures and open windows are capped at `300000` ms so a typo cannot make the circuit unreachable or leave valid traffic shed for hours. The upstream idle connection pool timeout is capped at `60000` ms and the idle per-host pool cap is capped at `4096`; `0` remains allowed only for `upstream_pool_max_idle_per_host` to disable idle origin pooling. Runtime filter reload reads are capped at `16777216` bytes, runtime/static rule caps are capped at `8192` rules, adaptive event-log files are capped at `1073741824` bytes before rotation, adaptive signature/path-shape windows are capped at `262144` each, HTTP tracked IPs are capped at `1048576`, HTTP tracked signatures/path shapes are capped at `262144` each, and TCP tracked IPs are capped at `1048576`, so a bad config cannot convert learned filtering or limiter state into unbounded memory or hot-path scans. The HTTP method allowlist must be non-empty, count-capped, token-valid, duplicate-free, length-bounded, and must not include unsupported tunnel/diagnostic names (`CONNECT`, `TRACE`, or `TRACK`) so generated `405 Allow` responses stay meaningful and startup cannot opt into unsupported reverse-proxy behavior. The optional Host allowlist must be count-capped and exact-entry only; entries must be trimmed, valid HTTP authorities without userinfo or wildcards, duplicate-free, and length-bounded by both the Host guard and the static host-entry cap. Client-IP trust config must use a bounded valid HTTP field name and a bounded, duplicate-free trusted-proxy list of valid IP/CIDR entries. Static filter rules must fit the static rule-count cap and every static rule must have bounded matchers plus a bounded `block` action with a 4xx/5xx status. Admin prefixes must be absolute, non-root path prefixes without query, fragment, whitespace, or control characters. Configured metrics tokens must be non-empty, non-blank, trimmed, free of control characters, and capped at 256 bytes.
- Apply `runtime.min_nofile` before listeners start. On Unix-like systems AlturaProt raises the process soft `RLIMIT_NOFILE` up to the configured floor when the inherited hard limit allows it, and fails startup if the configured floor cannot be met. This prevents a low service-manager or shell file-descriptor limit from silently undercutting configured connection caps.
- When `runtime.min_nofile` is enabled, validate the final soft NOFILE limit against the configured socket budget before binding listeners: fixed reserve, HTTP/TCP listener shards, HTTP downstream connections, HTTP upstream in-flight requests, HTTP idle upstream pool, and TCP downstream/upstream socket pairs. Configurations with unbounded `max_connections: 0` fail this capacity check because the descriptor budget is not knowable.
- Wait for SIGINT/Ctrl-C and, on Unix, SIGTERM. Both signals publish shutdown to listener tasks; `runtime.shutdown_grace_ms` defaults to `2000` ms and is used as one listener-drain deadline before process exit. Set it to `0` only for deliberately fast local benchmark children.

## HTTP Path

- Accept HTTP/1 connections with Hyper on listener sockets created with explicit `listen_backlog`. `http.accept_shards` can create multiple `SO_REUSEPORT` listener sockets for the same address so Linux can distribute connection accepts during churn while all shards share the same userspace limiter state.
- Resolve the client identity from the TCP peer, or from the configured forwarded client-IP header only when the peer is in a configured trusted proxy range. Trusted forwarded chains are capped by byte length and hop count, and malformed or excessive chains are rejected before signature, filter, rate-limit, or upstream work. When trusted identity comes from `X-Forwarded-For`, AlturaProt canonicalizes duplicate XFF field lines into the same ordered comma-separated chain used for client identity before forwarding upstream. When trusted identity comes from a different configured header, such as `CF-Connecting-IP`, that header must be a single non-empty IP value: duplicate field lines and comma lists fail closed. For accepted custom identity headers, AlturaProt strips inbound XFF and synthesizes a clean `resolved-client, peer` XFF chain instead of preserving spoofable XFF values that were not used for identity.
- Reject malformed, oversized, duplicate, or invalid trusted-proxy entries at config load, and reject global trusted-proxy ranges on non-loopback HTTP listeners so direct clients cannot rotate spoofed forwarded identities around per-client rate buckets.
- Build a stable request signature from method, normalized path, query shape, user-agent family, and accept header. Path-shape derivation collapses numeric, UUID, hex, long dynamic, and high-confidence short token segments while preserving obvious route versions such as `v1` and `v2`. Short all-lowercase two- and three-character segments stay distinct for normal routing precision, but also feed a bounded sibling-churn parent key used by the path-shape budget when many different short siblings appear under the same parent route.
- Observe the signature in a rolling adaptive detector with bounded signature and path-shape window maps. Signature activation uses token-bucket style counters rather than a tumbling one-second reset. Path-shape windows still track and sample observed route-family pressure for analyzer visibility, but broad path-shape filter activation requires strong evidence such as deterministic rate-limit, trusted-proxy aggregate, filter, or body-guard denial. There are no hardcoded benign route-family exemptions, and high-cardinality fresh signatures or path shapes cannot evict recent detector windows; full shards reclaim idle windows and otherwise stop admitting new detector keys while preserving existing evidence.
- Write adaptive attack events to JSONL through a bounded queue and dedicated writer thread. User-controlled event fields are truncated before enqueueing, so a flood cannot turn one event into an unbounded memory or disk payload. `adaptive.event_log_queue_capacity` is capped at `8192` events; request workers use nonblocking enqueue of owned events, and when the queue is saturated or the writer discards an event after an open/rotate/write failure, `altura_event_log_dropped` increments instead of backpressuring the hot path. JSON serialization, file writes, flushes, and rotation happen in the writer thread. The first event flushes immediately for analyzer visibility; high-rate bursts flush at most once per positive `adaptive.event_log_flush_interval_ms`, idle flush timeouts still drain batched events, Unix log files are created/fixed to owner-only permissions, and the log rotates to numbered backups before positive `adaptive.event_log_max_bytes` is crossed.
- Enforce allowed HTTP methods, default rejection of method-override headers unless `http.allow_method_override_headers` is enabled, Host header validity/allowlist, absolute-form request-target authority validation with only `http` and `https` schemes accepted, forwarded-header and client-IP/proxy identity header sanitization, trusted-forwarded-header byte/hop bounds, downstream HTTP/1 keep-alive disabled by default, initial raw HTTP/1 framing precheck, request framing validation for exposed `Content-Length` and `Transfer-Encoding` ambiguity, default rejection of chunked request bodies unless `http.allow_chunked_request_bodies` is enabled, default rejection of compressed request `Content-Encoding` unless `http.allow_compressed_request_bodies` is enabled, default `Expect` rejection unless `http.allow_expect_continue` is enabled, bounded `Range` requests before origin work, default removal of client `Accept-Encoding` before origin work unless `http.forward_accept_encoding` is enabled, header read timeout capped at `60000` ms, max header bytes, max header field-line bytes, max header count, startup-capped request-target/query/path segment limits including absolute-form scheme and authority bytes, default-stripped request trailers with opt-in capped forwarding and startup-capped trailer limits, max request body by `Content-Length` and streamed bytes with startup values capped at `1073741824` bytes, request-body idle timeout capped at `60000` ms, request-body recent-window minimum data-rate floor capped at `1048576` B/s with post-grace accounting measured from the grace deadline, max request count per connection capped at `10000`, and max connection duration capped at `3600` seconds. The raw initial precheck searches for the header terminator only across the newly appended bytes plus the required three-byte overlap, treats a terminator that arrives after `http.max_header_bytes` as oversized, and rejects any first-request header field above `http.max_header_line_bytes`; parsed keep-alive follow-up requests run the same field-line check before client-IP, Host, signature, filter, rate-limit, or upstream work.
- Apply per-peer and global HTTP connection-open token buckets before spawning the connection task, so short-lived churn that never reaches request parsing is still throttled. Obvious global connection saturation is checked before per-client state insertion, and per-client connection-open tokens are consumed only after the global connection-rate and active-connection gates also admit the open.
- Serve `GET` admin health and metrics endpoints only after per-client, signature, global, and trusted-proxy request rate checks; metrics require exactly one `x-altura-admin-token` header matching `http.admin_token`.
- Apply normalized request-signature, normalized path-shape, short-token sibling-churn, global, and per-client-prefix token-bucket limits with two-phase admission before filter evaluation, so over-budget traffic returns `429` before taking runtime/static filter read locks or scanning learned rules. A request denied by one rate dimension does not consume tokens from another dimension, and the global request bucket mutex is held only while checking or consuming that bucket, not across per-client shard cleanup. Client IPs are normalized with `http.limits.ipv4_prefix_len` and `http.limits.ipv6_prefix_len` before per-client request, connection-open, and upstream in-flight buckets, then hash-sharded so prefix-normalized IPv4 networks do not collapse into one low-bit shard; defaults keep IPv4 exact at `/32` and aggregate IPv6 by `/64`. The signature bucket catches one repeated method/path/query/header shape, while the path-shape bucket catches polymorphic floods that rotate long or high-confidence token segments inside the same route family. A separate bounded sibling-churn bucket uses the same `path_shape_rps`, `path_shape_burst`, and `max_tracked_path_shapes` settings to catch lowercase short-token rotation such as many distinct `/api/{ab,cd,ef}` siblings without blindly merging every stable short route. Short-token sibling denials are observed by the adaptive detector and event log using the parent descriptor such as `/api/:short-token`, and explicit filter rules with that `path_shape` can also match lowercase short siblings through the same parent descriptor. Strong adaptive path-shape activation is independent from the event sample cap, so throttled sample logging cannot suppress a learned block when strong evidence appears. When a trusted proxy supplies a forwarded client IP, also apply separate per-proxy aggregate request-rate and upstream in-flight caps keyed by the exact immediate peer IP so XFF rotation cannot bypass every client-prefix bucket or monopolize origin concurrency from one edge hop.
- Apply static and active adaptive filters only after the deterministic request-rate buckets admit the request. Loaded rules are count-capped, require at least one bounded matcher, require bounded block responses, keep validated header names and case-insensitive match needles in compiled form, and request path shape is cached once per evaluation so large learned rule sets do not repeatedly parse header names, allocate lowercase strings, or recompute the same derived path shape. Request workers clone the current rule-list snapshot under a short read lock, then scan rules without holding the rule-list lock; adaptive activation updates per-rule atomic deadlines, and reload swaps in a new snapshot while preserving active rule deadlines. This keeps learned-filter activation and reloads from blocking in-flight filter scans on a long-held writer lock.
- Bound per-client, signature, and path-shape limiter state with `max_tracked_ips`, `max_tracked_signatures`, and `max_tracked_path_shapes`. Rate buckets evict stale entries but deny new keys when a shard is full of active buckets, so high-cardinality rotation cannot reset an existing hot bucket by forcing oldest-active eviction. Connection and in-flight limiters evict stale or inactive entries and deny new entries only when a shard is full of active entries; globally saturated opens or in-flight requests are rejected before creating avoidable per-client state. Limiter cleanup probes are capped per insert attempt, so a new attacker-controlled key cannot force a full large-shard scan while holding a limiter mutex.
- Acquire a global/per-client in-flight upstream request permit before proxying. The permit is held until the upstream response body is dropped, so slow origin responses cannot consume unbounded proxy/origin capacity.
- Bound new upstream TCP connection attempts with `upstream_connect_timeout_ms` before the broader upstream response-header wait budget applies. `upstream_connect_timeout_ms` and `upstream_timeout_ms` are both capped at `60000` ms at startup so a bad config cannot pin origin in-flight permits behind minute/hour connect or response-header waits. Passive upstream failure circuits are keyed by normalized request path shape and share the `http.limits.max_tracked_path_shapes` state cap, so repeated failures in one route family do not fail-fast unrelated route families. A shape circuit opens after `upstream_failure_threshold` consecutive upstream connect/header failures or upstream response timeouts, capped at `1024`, then sheds matching origin-bound requests locally with `503`, `Retry-After: 1`, `Cache-Control: no-store`, and `Connection: close` for `upstream_failure_open_ms`, capped at `300000` ms. The shape circuit resets on a successful upstream response header for the same path shape and increments `altura_http_upstream_circuit_open` for locally shed open-circuit requests. Circuit state preserves currently open entries under high-cardinality churn; full shards reclaim closed or expired entries and otherwise skip tracking a new failing shape instead of evicting an open fail-fast route family.
- Bound the upstream HTTP client idle pool with `upstream_pool_max_idle_per_host` capped at `4096` and `upstream_pool_idle_timeout_ms` capped at `60000` so origin keep-alive sockets do not accumulate or linger without operator control. Set `upstream_pool_max_idle_per_host` to `0` only when intentionally disabling idle origin pooling.
- Rewrite the request to the upstream and stream the response back.
- Bound upstream response-header wait time, response header bytes/field-line/count, default-stripped response trailers with opt-in capped forwarding, response body idle time capped at `60000` ms, response body recent-window minimum data-rate floor capped at `1048576` B/s with post-grace accounting measured from the grace deadline, and response body size with startup values capped at `1073741824` bytes so a slow, stuck, malformed, or unexpectedly large origin response does not consume proxy/origin capacity indefinitely. Oversized declared upstream `Content-Length` is rejected before upstream success is recorded; unknown-length responses still use the streaming response guard.
- Wrap downstream writes with `downstream_write_timeout_ms`, capped at `60000` ms, so slow-reading clients cannot keep response streams and upstream in-flight permits pinned indefinitely.

## TCP Path

- Accept raw TCP connections with Tokio on listener sockets created with explicit `listen_backlog`. `tcp[].listen` and `tcp[].upstream` are validated at config load so malformed listener addresses or URL-shaped upstream typos fail before the proxy accepts clients. `tcp[].accept_shards` uses the same bounded `SO_REUSEPORT` sharding path as HTTP while sharing one connection limiter per TCP proxy.
- Apply per-client-prefix and global connection-open rate limits plus global and per-client-prefix concurrent-connection caps. Obvious global connection saturation is checked before per-client state insertion, and per-client connection-open tokens are consumed only after the global connection-rate and active-connection gates also admit the open. Raw TCP peer IPs are normalized with `tcp[].limits.ipv4_prefix_len` and `tcp[].limits.ipv6_prefix_len`; defaults keep IPv4 exact at `/32` and aggregate IPv6 by `/64`.
- Connect to the configured upstream with `tcp[].connect_timeout_ms`, capped at `60000` ms at startup so outbound TCP connect attempts cannot hold accepted connection permits for minute/hour waits after a bad deploy.
- Relay bytes bidirectionally through independent per-direction read/write halves with a shared connection-idle monitor, maximum connection duration capped at `3600` seconds, and optional per-direction recent-window minimum data-rate guards capped at `1048576` B/s that measure from the grace deadline once post-grace bytes arrive for raw TCP services where slow-drip payloads should be ejected before max duration. One write-backpressured direction does not stop the relay from reading and writing the opposite direction.

## Edge Path

AlturaProt's userspace proxy is the request-aware layer, not a replacement for network edge DDoS controls. Public deployments should run behind provider/CDN/scrubbing controls, a host firewall/sysctl profile, and a service-manager unit that preserves the proxy's resource assumptions. The templates in `ops/` provide host-level backstops for invalid conntrack states, dead UDP on protected TCP service ports, SYN pressure, ICMP flood pressure with essential ICMPv4/ICMPv6 control exemptions, fragment reassembly memory pressure, coarse connection storms, explicit finite nftables SYN-rate and connection-limit set capacity, service restart storms, excessive process/file-descriptor growth, and privilege containment. Generic protected-port TCP rules use `meta l4proto tcp` before transport-header matches so dual-stack invalid-flag, global SYN, clean-SYN, and global new-connection backstops match the real transport protocol. IPv6 host-edge SYN and connection-count keys aggregate source addresses by `/64`, matching the userspace default so interface-identifier rotation does not split across exact-address buckets before the proxy sees traffic. IPv6 TCP and ICMPv6 predicates use `meta l4proto` so extension headers do not skip those host-edge gates. Sysctl fragment reassembly bounds keep incomplete IPv4/IPv6 fragment queues from sitting in kernel memory for long periods during fragment floods.

## Filter Safety

Filters are allowlisted JSON rules. Provider output is sanitized before it reaches `runtime/filters.json`, and the Rust proxy only implements the supported condition/action fields. Static config filters and runtime filters share rule validation: non-empty IDs, at least one bounded matcher, no root-wide `path_prefix: "/"`, bounded method/header matcher counts, valid header names, `action.kind: "block"`, 4xx/5xx block statuses, and bounded block bodies. Runtime reloads reject non-regular files, oversized files, invalid rules, and files with too many runtime filters; reload failures keep the last good in-memory rules. Supported header and user-agent conditions are compiled on load so request evaluation stays allocation-light. Rule-list snapshots are immutable after publication except for per-rule atomic adaptive deadlines, so request evaluation, activation, and reload have narrow synchronization surfaces. Unsupported model output is ignored. Common-looking path shapes are not source-code exemptions; if a route family needs a different tolerance, tune its rate thresholds or learned/static filters explicitly.

Adaptive rules are dormant until their learned signature crosses the configured threshold. They then activate for a TTL and expire automatically.

## False Positive Strategy

- Prefer signatures over broad path or user-agent blocks.
- Keep static rules only for known hostile probes.
- Keep adaptive filters dormant until a burst threshold is crossed.
- Use short TTLs and event logs for auditability.
- Treat blackholing as an upstream emergency control, not as normal L7 filtering.
